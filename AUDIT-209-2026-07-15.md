# Poseidon v2 — Auditoria de segurança dirigida — #209 — 2026-07-15

Auditoria de três superfícies: **TLS/FFI OpenSSL**, **auth JWT/Digest** e
**superfície de DoS** (rate limit / timeout / body / circuit breaker / guard).
Cada achado foi provado por leitura do fonte (arquivo:linha + cenário concreto);
o que estava correto é declarado explicitamente. Correções aplicadas nesta sessão
marcadas **[CORRIGIDO]**; testes de regressão adicionados onde testável (Win64).

## TLS / OpenSSL FFI — `SSL.pas`, `SSL.Manager.pas`, `Security.pas`

Camada FFI **cuidadosamente escrita**. Cert/key/mTLS falham alto (sem downgrade
para plaintext); bytes de SNI/ALPN tratados sem path de decode-DoS e sem overread;
helpers puros (`IsIPInCIDR`, `IsPathSafe`, `HasRequestSmuggling`) fail-closed com
cobertura de teste. Achados (todos LOW/latentes, não corrigidos — mudança de FFI
exige validação de runtime nas 2 plataformas):
- **[LOW] `BIO_ctrl` (`SSL.pas:67`):** `larg`/retorno declarados `Integer`; C é `long`
  (64-bit no Linux64). Não explorável hoje (único uso `larg=0`); latente se
  reusado. Fix: `NativeInt`.
- **[LOW] `SSL_CTX_ctrl` (`SSL.pas:68`):** retorno `NativeInt` mas `long`=32-bit no
  Win64; upper dword indefinido. Inócuo (retorno ignorado); latente se algum
  callsite passar a checar `<> 0`.
- **[LOW] SNI dict race (`SSL.Manager.pas:66-71` vs `150-181`):** `TDictionary` sem
  lock, só perigoso se `AddSSLCert` for chamado com o servidor no ar. Fix: lock ou
  documentar "configure certs antes de Listen".
- **[LOW] ALPN fallback (`SSL.pas:483-492`):** retorna OK selecionando o 1º protocolo
  do cliente mesmo sem overlap; RFC 7301 pede `no_application_protocol` fatal. Não
  é memory-unsafe.
- **NOTA (fora de escopo):** a máquina de estados TLS de runtime (WANT_READ/WRITE,
  drain de BIO parcial, renegociação, lifetime de `SSL*` cross-thread) **não** está
  nesses 3 arquivos — vive em `Poseidon.Net.HttpServer` (`_ProcessRecvSSL`/
  `_EncryptAndSend`) e precisa de um passo dedicado antes de declarar #209 fechado.

## Auth JWT — `Middleware.JWT.pas`

**Sólido.** `alg:none` bloqueado (alg fixado em HS256 antes de verificar);
assinatura verificada ANTES de confiar em claims; `ConstantTimeEquals` sem
short-circuit; assinatura vazia rejeitada; base64url pré-auth (header) em try/except
+ UTF-8 guardado; exp/nbf com semântica UTC correta. Único gap:
- **[LOW] Sem `aud`/`iss` + `exp` opcional:** com secret HMAC compartilhado entre
  serviços, um token de A é aceito por B (replay cross-service); token sem `exp`
  vale para sempre. Recomendação: params opcionais `aud`/`iss` e opção de exigir
  `exp`. (Não corrigido — é enhancement de API, não bug.)

## Auth Digest — `Middleware.Digest.pas`

Partes criptográficas corretas (compare constant-time, nonce emitido pelo servidor
+ `nc` estritamente crescente = anti-replay, `qop` fixado, cap `CMaxNonces`).
Corrigidos:
- **[MEDIUM CORRIGIDO] uri não vinculado ao path (`:286`):** HA2 usava o `uri` do
  header sem compará-lo ao alvo real; um header capturado para `/public` podia ser
  redirecionado para `/admin` (o digest ainda fecha, pois ambos usam o `uri` do
  header) → escalada onde a autorização é por-path. **Fix:** rejeita 401 se o path
  do `uri` do header ≠ `ACtx.Path`. Regressão: `UriMismatch_Returns401`.
- **[MEDIUM CORRIGIDO] `Purge` O(n) por request sob lock global (`:124`):** rodava a
  cada request (inclusive não-autenticadas, que emitem nonce) → um flood de 401
  serializa todo o Digest num scan de 100k sob um único lock. **Fix:** `Purge`
  amortizado (≤1×/30s via `FLastPurge`) + expiry inline por-lookup no `Accept`
  (rejeita nonce expirado mesmo sem sweep). O(1) amortizado por request.
- **[LOW risco aceito] Enumeração de usuário por timing (`:280`):** usuário
  desconhecido retorna antes do compare. Um "dummy HA1" para uniformizar timing
  seria **pior** (se previsível, vira bypass de auth); manter o retorno cedo é a
  escolha segura. Documentado, não corrigido.

## Superfície de DoS — RateLimit / Timeout / Guard / BodyLimit / CircuitBreaker

- **[HIGH CORRIGIDO] Mapa do RateLimit sem teto (`RateLimit.pas:48,91-123`):** a
  única reclamação é um sweep a cada 60s; entre sweeps o mapa cresce 1 entrada por
  chave distinta. Rotação de origem IPv6 (/64 = 2^64) ou XFF spoofado → OOM, **e**
  bypass (toda req vira "novo IP", contador reseta). **Fix:** teto rígido
  `AMaxTrackedKeys` (default 100k, configurável) + sweep de emergência amortizado
  (≤1×/5s) + fail-closed O(1) (429) quando cheio. Regressão:
  `DistinctKeysBeyondCap_Refused`.
- **[MEDIUM CORRIGIDO] Chave XFF spoofável/sem limite (`RateLimit.pas:82-84`):**
  primeiro token do XFF usado verbatim, multi-KB → chave gigante + cardinalidade
  ilimitada. **Fix:** token XFF vazio ou > 45 chars (máx IP textual) → cai de volta
  ao peer real.
- **[MEDIUM documentar] Timeout não cancela (`Timeout.pas:52-56`):** é um cronômetro
  em volta de `ANext()`; o handler roda até o fim, só o 504 é aplicado depois. N
  requests lentas ainda esgotam o worker pool. É política, não proteção — a defesa
  real precisa estar na camada de I/O/worker.
- **[MEDIUM documentar] BodyLimit tardio (`BodyLimit.pas:23`):** checa
  `Length(RawBody)` que o parser JÁ materializou; o cap efetivo é o
  `MaxRequestSize` do core (esse sim streaming). A comparação em si é segura
  (`>` estrito, `Integer`→`Int64`). Documentar que proteção = `min(BodyLimit,
  MaxRequestSize)`.
- **CORRETOS:** CircuitBreaker (toda mutação sob `FLock`; half-open admite 1 probe;
  ring de 60 slots) e Guard (allow-list + `IsPathSafe` + smuggling antes de
  `ANext`) — sólidos. Aritmética de janela do RateLimit sem overflow prático.

## Veredito

Nenhum CRITICAL; nenhum HIGH em TLS/FFI ou JWT. O único HIGH (mapa do RateLimit)
e os dois MEDIUM do Digest — os achados exploráveis por um atacante ativo — foram
**corrigidos com regressão** (Win64). Restam: itens de FFI latentes (mudança de
tipo, exige validação Linux), a natureza não-preemptiva de Timeout/BodyLimit (por
design; documentar), o enhancement `aud`/`iss` do JWT, e — o maior gap remanescente
— a **máquina de estados TLS de runtime no HttpServer**, que está fora destes
arquivos e precisa de um passo dedicado antes de fechar #209.
